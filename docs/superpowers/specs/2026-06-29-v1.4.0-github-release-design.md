# OmniPlayer v1.4.0 GitHub Release Design

## Goal

Publish OmniPlayer v1.4.0 as a polished Windows integration workflow for spatial-audio playback, monitoring, and AoIP output, while describing its relationship to upstream components accurately.

## Homepage positioning

The README will lead with a concise bilingual description:

- OmniPlayer integrates Omniphony Studio, mpv-omniphony, `orender`, and a compatible bridge into one Windows workflow.
- It addresses path configuration, multi-process startup, OSC monitoring, audio-device selection, and AoIP output through ASIO.
- It gives non-specialist Windows users a more approachable way to play and monitor supported spatial-audio media.
- Decoding, playback, bridge, and rendering capabilities come from independently licensed upstream components. OmniPlayer does not claim or implement proprietary codec technology.

## Homepage presentation

- Retain the OmniPlayer product icon and title.
- Replace the old product screenshot with the supplied v1.4.0 interface screenshot.
- Place the screenshot directly after the bilingual introduction so visitors can understand the product before reading technical details.
- Update the feature list to reflect the v1.4.0 interface, English/Chinese switching, portable runtime packaging, OSC supervision, diagnostics, and ASIO/WASAPI selection.

## Release

- Update `RELEASE_NOTES.md` for v1.4.0.

- Create the public GitHub release tag `v1.4.0`.
- Upload `OmniPlayer-Portable-v1.4.0.zip` and publish its SHA-256 checksum in the release notes.

## Verification

Before publishing:

- Build the launcher in Release mode.
- Run the smoke-test project.
- Verify the portable archive contains the launcher, mpv, Studio, `orender`, bridge, guide, and legal notices.
- Confirm the new screenshot renders from the README path.
- Review the staged diff so generated work files and unrelated artifacts are not committed.
